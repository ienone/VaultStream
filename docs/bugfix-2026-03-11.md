## Issue #1 — 小红书登录状态未能正常识别

### 问题描述

通过扫码登录小红书后，`check_platform_status` 接口仍然返回未登录状态。原因是检测逻辑仅依赖单一 CSS 选择器 `.user.side-bar-component`，在带 Cookie 冷启动访问时该元素可能未加载或已变更，导致误判。

### 修复方案

- 为小红书平台配置增加多个备选选择器 `check_selectors` 和关键 Cookie 名称 `auth_cookie_names`（`a1`、`web_session`）。
- `_check_status_pw_job` 增加 `networkidle` 等待，优先检查 auth cookie 是否存在于当前上下文中，命中即返回 `True`。
- 新增 `_has_any_selector` 辅助方法，遍历多个选择器，任一命中即视为已登录。

### 修改文件

| 文件 | 改动 |
|------|------|
| `backend/app/services/browser_auth_service.py` | 更新平台配置；新增 `_has_any_selector`；重写 `_check_status_pw_job` 检测逻辑 |

## Issue #2 — 登录 Cookie 未传递给专用解析器

### 问题描述

通过浏览器扫码登录后，Cookie 被正确保存到数据库（`set_setting_value("xiaohongshu_cookie", ...)`），但 `XiaohongshuAdapter` 在初始化时从 `settings.xiaohongshu_cookie` 读取，后续解析请求可能仍使用旧值。同时 `delete_setting_value` 删除 Cookie 后未清空内存中的全局 `settings` 对象，导致注销后旧 Cookie 残留。

### 修复方案

- 在 `XiaohongshuAdapter.parse()` 方法开头调用新增的 `_refresh_cookie_from_settings()`，每次解析前从 DB 缓存获取最新 Cookie。
- `delete_setting_value` 删除配置后同步清空全局 `settings` 对象对应字段。

### 修改文件

| 文件 | 改动 |
|------|------|
| `backend/app/adapters/xiaohongshu.py` | 新增 `_refresh_cookie_from_settings()`，在 `parse()` 首行调用 |
| `backend/app/services/settings_service.py` | `delete_setting_value` 中增加全局 settings 对象同步清空逻辑 |

## Issue #3 — Telegram Bot 配置后未能正常启动

### 问题描述

引导页配置 Telegram Bot 后，后端调用 `restart_telegram_bot` 启动子进程。`start_telegram_bot` 内部使用 `asyncio.run(get_setting_value("api_token"))` 获取 API Token，在某些场景下可能因已有事件循环而失败（`RuntimeError: This event loop is already running`），导致子进程无法获得正确的 API Token 而启动失败。

### 修复方案

- `start_telegram_bot` / `restart_telegram_bot` 增加可选参数 `api_token`，由调用方在 async 上下文中预先获取后显式传入。
- `_sync_telegram_bot_process`（路由层调用入口）在 async 环境中安全获取 token 后传递给线程池任务。
- 保留 `asyncio.run` 作为 CLI 直接调用时的兜底方案。

### 修改文件

| 文件 | 改动 |
|------|------|
| `backend/app/services/telegram_bot_service.py` | `start_telegram_bot` / `restart_telegram_bot` 新增 `api_token` 参数 |
| `backend/app/routers/bot_config.py` | `_sync_telegram_bot_process` 预获取 token 并传入 |

## Issue #4 — QQ Bot 管理员 ID 预填 Telegram 的值

前端引导页和设置页中，QQ Bot 的管理员/白名单/黑名单 ID 全部读写 `telegram_admin_ids` 等 key，导致：

- 移除设置页中重复覆盖 `telegram_admin_ids` 的代码块。

### 修改文件

| 文件 | 改动 |

## Issue #5 — 自动获取 Bot 加入的群列表

### 问题描述

用户配置完 Bot 后需要手动同步群列表，增加了配置负担。

### 修复方案

- **QQ (Napcat)**：新增 `_auto_sync_chats_background` 后台任务，在 `create_bot_config` / `update_bot_config` 成功后自动调用 Napcat `/get_group_list` 接口拉取并 upsert 群列表。
- **Telegram**：API 不支持主动列出 Bot 加入的所有群组，继续依赖 `ChatMemberHandler` 事件驱动自动发现（Bot 被拉入群/权限变更时自动上报）。启动时会遍历已知 chat 刷新元信息。

### 修改文件

| 文件 | 改动 |
|------|------|
| `backend/app/routers/bot_config.py` | 新增 `_auto_sync_chats_background()`；`create_bot_config` / `update_bot_config` 中触发后台同步 |

## Issue #7 — 手机端探索页操作报 301 错误

### 问题描述

手机端对探索中的内容执行"收藏"或"移除"操作时报错：

```
DioException [bad response]: status code 301
RequestOptions.validateStatus was configured to throw for this status code.
```

原因：Dio 默认 `followRedirects: true`，遇到 301 时会将 PATCH/POST 请求转为 GET 重发，导致请求体丢失、服务端返回异常。该 301 通常由 Nginx 的 URL 规范化（如 trailing slash、HTTPS 重定向）触发。

### 修复方案

- Dio `BaseOptions` 设置 `followRedirects: false`，阻止自动跟随重定向。
- 设置 `validateStatus` 接受 `< 400` 的状态码，3xx 不再抛异常。
- 建议同步检查 Nginx 配置，确保对 API 路径使用 `proxy_pass` 而非 301 重定向，或至少使用 308 保留请求方法。

### 修改文件

| 文件 | 改动 |
|------|------|
| `frontend/lib/core/network/api_client.dart` | `BaseOptions` 增加 `followRedirects: false` 和 `validateStatus` |

## Issue #8 — 收藏库不会自动刷新促进的内容

### 问题描述

在探索页将内容"收藏到收藏库"（promote）后，收藏库页面不会自动刷新展示新内容。原因：

- 后端 `update_discovery_item` / `bulk_action` 修改了 `discovery_state` 但未发布任何 SSE 事件。
- 前端收藏库 `CollectionNotifier` 监听 `content_created` / `content_updated` / `content_deleted` 事件来增量更新，但 promote 操作没有产生这些事件。

### 修复方案

- `update_discovery_item` 在 `db.commit()` + `db.refresh()` 后发布 `content_updated` SSE 事件。
- `bulk_action` 在 `db.commit()` 后对每条内容发布 `content_updated` 事件。
- 前端收藏库已有对 `content_updated` 的增量处理逻辑（通过 SSE EventBus + 300ms 防抖），无需额外修改。

### 修改文件

| 文件 | 改动 |
|------|------|
| `backend/app/routers/discovery.py` | 导入 `event_bus`；两个端点中增加 `event_bus.publish("content_updated", ...)` |
